# Example: `act` / `act [需求]`

这份样例定义 `act` 的标准展示方式。

## 目标

让任何 AI 都把 `act` 保持成轻量在线推荐，而不是第二份体检报告。

## 标准输出骨架

## 裸跑 `act`

### 1. 当前身份

```text
目标宿主：Claude Code
已装技能：17 个
主题关键词：sales ai content knowledge
```

### 2. 本地候选建议

| 来源 | 说明 |
|---|---|
| WorkBuddy | 87 个你还没有；适合先从本机已验证来源里挑能力 |
| CC-Switch | 21 个你还没有；和当前宿主重叠度高，适合先做路线体检 |

### 3. 追问一句

```text
你现在更想补哪类能力？直接回一句：飞书协作 / 记忆知识库 / 搜索浏览器 / 图像内容 / 还不确定
```

### 4. 下一步推荐

```text
直接回答上面那句；回答后再展开在线发现雷达
```

## 带需求的 `act [需求]`

### 1. 当前身份

```text
目标宿主：Claude Code
已装技能：17 个
主题关键词：sales ai content knowledge
当前问题/需求：飞书知识库
```

### 2. 本地候选建议

| 来源 | 说明 |
|---|---|
| WorkBuddy | 87 个你还没有；适合先从本机已验证来源里挑能力 |
| CC-Switch | 21 个你还没有；和当前宿主重叠度高，适合先做路线体检 |

这里最好再补一句：

`如果现在只试 1 个，先看 WorkBuddy -> agent-mail，因为它更贴近当前主题，而且偷过去前置成本低。`

### 3. 在线发现雷达

#### 商店 / 官方入口建议

| 入口 | 适合场景 |
|---|---|
| OpenCode Skills / Rules | 跨宿主技能与规则设计 |
| OpenClaw 生态技能榜 | 找现成跨宿主技能再做兼容性检查 |
| Claude Code 官方概览 | 核对项目规则与宿主行为 |

#### 在线候选建议

| 技能 | 作者 | 星数 | 说明 |
|---|---|---:|---|
| creating-sales-enablement | amogha-dalvi | 2 | 销售赋能内容 |
| write-sales-copy | sogadaiki | 0 | 销售文案 |

热度判断一句话就够：

`在线候选整体热度一般，建议当灵感清单，优先核对本地库和官方入口。`

#### GitHub 需求雷达（如果用户给了明确需求）

```text
- 按当前需求搜 GitHub 仓库，只把它当探索项
- 如果没搜到明显相关结果，就明确说“先以本地候选和 SkillsMP 为主”
```

#### 趋势雷达（如果装了 github-trending-cn）

```text
- 最近 GitHub 上有几个新 repo 在冒头，但这里只当探索项
- 真要装，仍然回到 check / steal 做判断
```

### 4. 下一步推荐

只给一个最顺手的下一步：

```bash
check CC-Switch
```

## 禁止事项

- 不要再做一遍完整体检
- 用户没给需求时，不要第一反应就丢在线热榜大列表
- 不要把低星候选硬推成首选安装
- 不要跳过本地候选，直接把在线结果当主战场
- 不要一次给很多入口、很多候选、很多命令
