# decisions

这份文件记录 `skill-manager` 为什么这样设计，避免以后迭代时失忆。

## 2026-03-24 命令边界收口

### 决定

- `scan` 只负责发现本地库和候选来源
- `steal` 只负责迁移
- `check` 成为主命令，负责当前库健康度和路线判断
- `act` 收缩为身份驱动的轻量在线推荐

### 原因

- 用户真正常问的是“我现在这库怎么样，接下来补什么”
- 这更像 `check`，不是两套命令来回切
- `act` 保留，但不再做第二套体检系统

## 2026-03-24 输出要更确定

### 决定

- 脚本输出是事实源
- Claude / 其他 AI 不允许重算数量，不允许模糊改写
- `scan / steal / check / act` 都有固定输出边界

### 原因

- 真实测试里，多次出现“脚本是对的，模型二次总结讲偏了”
- 需要把“怎么说”也产品化，而不是只产品化脚本

## 2026-03-24 路径与上下文优先级

### 决定

- 用户给绝对路径时，路径优先
- 项目级 skill / 规则文件优先于主目录
- 主目录 skill 视为资源池，不直接等于项目已安装

### 原因

- 真实使用里，模型多次在用户给了精确路径后仍去别处乱找
- Claude Code 的项目级 / 全局层级容易让总结层归因混乱

## 2026-03-24 在线推荐降噪

### 决定

- `act` 必须短
- 先给官方入口，再给在线候选，再给一个下一步
- 低热度在线候选只能当灵感，不强推

### 原因

- `SkillsMP` 有搜索价值，但不稳定等于高置信推荐榜
- 用户更需要“下一步去哪看”，而不是一长串在线条目

## 2026-03-24 兼容 Windows，但不牺牲 macOS

### 决定

- 做增量兼容，不做扫描降级
- macOS 保留当前多来源本地扫描
- 原生 Windows 更保守，优先复制而不是软链

### 原因

- 当前产品优势之一就是本地扫描广
- 为兼容 Windows 把 macOS 能力削弱，会得不偿失

## 2026-04-07 智能和治理分层

### 决定

- Agent 负责智能判断、比较、推荐
- `skill-manager` 优先负责稳定事实、规则边界、治理依据
- 这次 MVP 先不追求更聪明的 `act`
- 先让 `scan / check / steal` 产出可复用状态文件
- 状态文件不是“写了就算”，后续追问必须 state-first

### 原因

- 如果 `skill-manager` 继续和 Agent 抢“谁更聪明”，边界会重新变糊
- 真实长期价值不在临场聪明，而在可复盘、可承接、可传承
- 先有状态底座，后面 Agent 才能稳定接着做解释和建议
- 如果 Agent 不先消费状态文件，用户体感上几乎不会觉得这次 MVP 有变化

### 这次落地

- `~/.skill-manager/state/latest-scan.json`
- `~/.skill-manager/state/latest-health.json`
- `~/.skill-manager/state/history.jsonl`
- `SKILL.md` / `output-contract.md` 明确了 state-first follow-up 规则
- `check <来源>` 的健康快照开始额外记录路线级摘要和候选名单

## 2026-04-07 GitHub 安装进入原生链路

### 决定

- `steal` 开始支持直接接收 GitHub 仓库 / tree URL
- 从 GitHub 安装时写入最小上游元信息
- `check` 在发现 GitHub 来源元信息后，顺手检查是否落后于上游

### 原因

- 真实使用里，用户已经自然在对话里发 GitHub URL 让 Agent 安装
- 如果这条路径一直靠 Agent 手动 `git clone + cp`，`skill-manager` 就只像旁观者
- 最小 MVP 不需要完整 marketplace，但需要把“开放来源安装 + 知道自己落后没”收进原生链路

### 这次落地

- `steal <GitHub URL>` 直接安装到目标库
- 安装后写入 `.skill-manager-source.env`
- `check` 输出 GitHub 上游状态，并写进 `latest-health.json`
- `history.jsonl` 默认自动裁剪，避免状态层无限增长

## 2026-04-07 双入口与一键体验

### 决定

- 空调用不再直接丢四命令表格
- 改成先给两个入口：`一键体验` / `命令模式`
- `一键体验` 不是第五个正式命令，只是组合流程
- 组合流程固定为：`scan -> check -> act -> steal preview`

### 原因

- 用户对 `scan / steal / check / act` 的概念天然是模糊的
- 先逼人理解命令，再体验价值，门槛太高
- 但如果默认直接自动跑，又会把选择权从人手里拿走
- 最平衡的方式是：先给人类一个选择，再让 Agent 去编排体验

### 这次落地

- 空调用脚本时，先给双入口菜单
- `一键体验` 默认只读，不直接安装 skill
- `命令模式` 保留老一套四命令，不做大改

## 2026-04-07 act 改成双通道推荐

### 决定

- `act` 继续保持轻量
- 但推荐不再只盯在线结果
- 改成同时给：
  - 本地候选建议
  - 在线发现雷达
- 如果本机装了 `github-trending-cn`，自动把它接成额外发现源

### 原因

- 真正最容易落地的 skill，很多时候不在大排行榜上，而在本机已经扫到的来源里
- 本地来源往往已经被别的宿主、项目或团队验证过，成功率更高
- 在线来源更适合补新线索，不适合直接替代本地候选

### 这次落地

- `act` 新增 `本地候选建议`
- `act` 把 `商店 / 官方入口建议`、`在线候选建议`、`趋势雷达` 收成 `在线发现雷达`
- `github-trending-cn` 只作为探索项来源，不直接越过 `check / steal`

## 2026-04-07 一键体验内部收口 + 问题驱动推荐

### 决定

- 给脚本加一个内部 `experience` 子命令，专门收口一键体验
- `act` 继续以身份为底座，但允许用户直接给一句需求做重排
- 如果用户给了需求，再补一层 GitHub 需求搜索；`github-trending-cn` 继续只做热榜探索

### 原因

- 只靠提示词编排一键体验，Agent 还是会临场发挥、重算数字、自己补动作
- 只靠身份推荐，容易把“我现在想找什么”这个强信号丢掉
- 只靠热门趋势，无法满足“我不是想看热榜，我是想按问题找东西”这类场景

### 这次落地

- 新增内部命令：`bash scripts/skill-mgr.sh experience [可选需求]`
- `SKILL.md` 明确要求一键体验优先走内部命令，不再手动串四个命令
- `act [需求]` 现在会同时重排：
  - 本地候选
  - SkillsMP
  - GitHub 需求雷达
  - github-trending-cn 趋势雷达

## 2026-04-08 act 改成先追问，再外找

### 决定

- 裸跑 `act` 时，不再第一时间展开在线热榜
- 先给当前身份判断和本地候选
- 再追问一句短问题，等用户回答后再走问题驱动推荐

### 原因

- 直接丢 GitHub / SkillsMP 热门榜，看起来像机器在报表，不像人在帮你选
- 很多时候用户自己并没有先说清“现在想补什么能力”
- 先追问一句，再去外面补线索，会更有人感，也更准

### 这次落地

- `act` 裸跑时改成 question-first
- `act <需求>` 继续保留完整在线发现雷达
- GitHub / SkillsMP 在线层只在用户给出需求后再展开
