# output contract

这份文件定义 `scan / steal / check / act` 的固定输出边界。

补充：`bind` 是高级修复入口，只在用户已经手动装好 GitHub skill、又想补来源登记时使用。

## state-first follow-ups

命令刚跑完时，后续追问先读状态文件，再做中文总结。

- `scan` 后优先读 `~/.skill-manager/state/latest-scan.json`
- `check` 后优先读 `~/.skill-manager/state/latest-health.json`
- `steal` 后优先读 `~/.skill-manager/state/history.jsonl`

只有状态文件缺失、明显过期，或者用户问的是状态里没有的细节，才回退到新的目录扫描、`ls`、`find`、`grep` 或额外联网探测。

## empty invocation

空调用时先只做一件事：

1. 给用户两个入口：`一键体验` / `命令模式`

禁止：

- 直接自动跑 `scan`
- 直接丢四命令表格
- 一上来讲一堆理念和教程

## one-click experience

一键体验只做四件事：

1. `scan`
2. `check`
3. `act`
4. `steal preview`

实现上优先直接调用内部收口命令：

- `bash <skill-dir>/scripts/skill-mgr.sh experience [可选需求]`

禁止：

- 直接安装 skill
- 输出成四大段原始工具日志
- 把它包装成第五个正式命令
- 手动串四个命令后再自己重算数字

## scan

只做三件事：

1. 整体总表
2. 当前目标库结构
3. 最值得看的少数来源

禁止：

- 把所有来源全部展开
- 重算脚本已经给出的数量
- 用模糊比较词替代事实

## steal

只做四件事：

1. 说明从哪里迁到哪里
2. 说明是软链还是复制
3. 如果来源是 GitHub，说明跟踪的是哪个 repo / 子目录 / 提交
4. 说明是否成功并给一个紧邻的下一步

禁止：

- 自动发明 `check <skill-name>`
- 失败后偷偷造依赖并伪装成原版
- 把“已复制文件”说成“已经稳定可用”

## bind

只做四件事：

1. 说明正在给哪个本地 skill 补溯源
2. 说明绑定到哪个 GitHub repo / 子目录
3. 说明安装提交是否能确认
4. 给一个紧邻的下一步 `check`

禁止：

- 把 `bind` 讲成重装
- 在没确认来源时伪装成精准版本管理
- 补登记后不告诉用户来源 URL

## check

只做四件事：

1. 当前库 / 路线是谁
2. 整体健康度
3. 发现的问题（包括已登记上游的 GitHub skill 是否落后）
4. 下一步建议

禁止：

- 扩写成冗长大报告
- 把静态问题说成实机问题
- 自己补不存在的命令模式

## act

只做四件事：

1. 当前身份
2. 本地候选建议
3. 在线发现雷达（含商店 / 官方入口建议、在线候选建议，必要时加趋势雷达）
4. 下一步推荐

如果用户直接给了需求句子，也允许在“身份打底”的前提下做问题驱动重排。

如果用户没有给需求句子，则先做：

1. 当前身份
2. 本地候选建议
3. 一句很短的追问
4. 下一步让用户直接回答需求

这时先不要展开 SkillsMP / GitHub 热榜大列表。

禁止：

- 再做一遍完整体检
- 把低星结果硬推成首选
- 跳过本地候选，直接把在线结果当主战场
- 给太多并列下一步

## next step rule

每次只主推一个下一步。

- `scan` 后优先推 `check <来源>`
- `steal` 后优先推 `check`
- `check` 后优先推 `act` 或一个最相关来源
- `act` 后优先推一个最相关本地动作
